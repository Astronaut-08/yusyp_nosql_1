# Налаштування оточення та запуск
1. Завантажте [датасет](https://www.kaggle.com/datasets/maharshipandya/-spotify-tracks-dataset)

2. Зареєструйтеся на [mongodb.com/atlas](mongodb.com/atlas).

3. Створіть кластер — виберіть провайдера AWS та регіон, найближчий до вас.

4. У розділі Database Access створіть користувача з паролем.

5. У розділі Network Access додайте свій IP (або 0.0.0.0/0 для розробки).

6. Перейдіть в створений кластер, натисніть Connect → Drivers, скопіюйте рядок підключення вигляду mongodb+srv://... — він знадобиться у скриптах. (обов'язково вкажіть пароль користувача в посиланні де це вимагається)

7. Завантажений датасет розмістіть в кореневій папці проекту.

8. В папці проекту створіть файл із назвою ".env" (саме з такою назвою), в цьому документі збережіть такий рядок: `MONGO_URI=посилання скопійоване з пункту 6 із паролем користувача`

9. Створіть віртуальне середовище в кореневій папці командою `python -m venv .venv`, після чого активуйте його командою:
- `.venv/Scripts/Activate.ps1` - для powershell
- `.venv/Scripts/Activate.bat` - для cmd
- `source .venv/bin/activate` - для bash

та встановіть всі необхідні залежності командою `pip install -r requirements.txt`

10. Запустіть скрипт `01_load_data.py` який знаходиться в папці *scripts*. 

11. Після цього на сайті через вкладку проект з'єднайтесь з базою даних або в MongoDB Compass (десктопна версія) додайте з'єднання і підключіться до бази і виконайти в MongoDB Shell скрипт із файлу `02_transform.js`

12. Вуаля, можна виконувати всі доступні скрипти з даним датасетом!

# Частина 1

### 1. Чому аудіо-характеристики винесені в окремий об’єкт audio_features, а не зберігаються плоско? Коли таке вкладення вигідне, а коли створює проблеми?


Аудіо-характеристики винесені в окремий об'єкт задля логічної ізоляції від метаданих(імена і тд.), що робить структуру інтуітивно зрозумілою. Окрім зрозумілості, також при роботі з фронтом зручніше оперувати із згрупованими об'єктами, а не визбирувати окремі поля із загального об'єкта.

**Таке вкладення вигідне тим, що:**
- за одне звернення до диска можна отримати зразу цілий об'єкт
- можна оновити весь блок характеристик за одну дію

**Дане вкладення створює проблеми тоді, коли:**
- існує можливість непердбачуваного зростання, наприклад дане вкладення містить коментарі користувачів, з часом вкладення досягне ліміту 16МВ, і тоді треба буде робити новий об'єкт щоб зберігати нові коментарі, що само по собі є редфлагом, або видаляти попередні коментарі, що теж не є хорошою практикою
- погано для частих оновлень, тому що потрібно уважно і чітко прописувати шлях через крапку, щоб випадково не видалити решту характеристик яких команда не стосувалась


### 2. Чому виконавці зберігаються як масив, а не як рядок? Які запити стають простішими?


В MongoDB працювати з масивами дуже зручно. Масиви дають можливість швидко проводити пошук, робити індексацію тощо. Якби дані зберігались як рядок, довелося б використовувати регулярні вирази і враховувати кожного разу багато нюансів, таких як послідовність, коми, пробіли і тд..
Запити пошуку спрощуються в рази, наприклад умовне використання (наприклад треба перевірити чи є артист в масиві, чи наприклад декілька артистів одночано є в масиві). Також по масивах легко робити аналітику за допомогою агрегації пайплайнів, що не було б можливим з рядками


### 3. Що таке $out і чим він відрізняється від $merge? Коли використовувати кожен?

`$out` створює нову колекцію на основі тієї з якої брались дані, чи перезеписує колекцію, якщо колекція з такою назвою яка вказана в `$out` вже існує, він видаляє її і перезаписує в нову. `$merge` оновлює існуючу колекцію, не видаляючи старі дані. `$out` потрібно використовувати тоді коли потрібно повністю замінити всі дані новити, він під це оптимізований і виконає цю дію набагато швидше, або наприклд коли потрібно вивантажити дані в якусь окрему колекцію тимчасову, для тестів. `$merge` краще використовувати для оновлення конкретних записів, чи додавання нових до існуючої колекції, наприклад в магазині змінилась ціна на один товар, чи додали нову категорію товарів до існуючих, ця робота для `$merge`.


# Частина 2

### 1. Для чого використовується інструкція $unwind?

Дана інструкція розгортає масив перетворючи його на окремі документи. Наприклад дивлячись на нашу базу пісень, є пісні які співали два артиста в тандемі, `$unwind` розбиває цей трек, на два документа, в одному документі дані пісні і всього лиш один співак (з двох які були першопочатково), в іншому документі точно всі такі ж дані пісні окрім співака, він буде зазначеним іншим. Якби було 3 співака то розбилось би на 3 документа і тд.

### 2. Чим $stdDevPop відрізняється від $stdDevSamp?

Обидва оператора рахують середнє відхилення, відмінність тільки в тому що: 
- `$stdDevPop` - рахує середнє відхилення для всієї генеральної сукупності які потрапили в aggregation stage.
- `$stdDevSamp` - рахує середнє відхилення конкретної вибірки

Варто зауважити що дані оператори треба застосовувати в залежності від призначення, так як `$stdDevSamp` враховуює потенційну похибку

# Частина 4

### 1. Що змінилося в плані виконання?

Операція пришвидчилась в сотні і більше разів, запит перестав читати весь документ, а тільки індексовані частини, наприклад в завданні *4.1* для пошуку *354* документів без індексу запит прочитав всі *113 999* документів, а з індексом тільки *354*, тобто рівно стільки скільки збігів існує.

### 2. Як зрозуміти, що індекс використовується? Наведіть скріншот або значення полів із `explain()`, які це підтверджують.

Щоб зрозуміти що індекс використовується ми дивимось на певні поля, а саме:
- **executionStats.nReturned** - кількість повернутих документів
- **executionStats.executionTimeMillis** - час виконання запиту
- **executionStats.totalDocsExamined** - кількість прочитаних документів
- **executionStats.totalKeysExamined** - кількість перевірених ключів (якщо 0 то прочитано кожен документ що є в базі)

**Приклади із завдання 4.1:**

*До індексування:*
```js
{
    nReturned: 354,
    executionTimeMillis: 120,
    totalKeysExamined: 0,
    totalDocsExamined: 113999,
}
```
*Після індексування:*
```js
{
    nReturned: 354,
    executionTimeMillis: 2,
    totalKeysExamined: 354,
    totalDocsExamined: 354,
}
```

**Приклади із завдання 4.2:**

*До індексування:*
```js
{
    nReturned: 1,
    executionTimeMillis: 72,
    totalKeysExamined: 0,
    totalDocsExamined: 113999,
}
```
*Після індексування:*
```js
{
    nReturned: 1,
    executionTimeMillis: 3,
    totalKeysExamined: 515,
    totalDocsExamined: 1,
}
```

## Завдання 3. Покривний запит
### Чи є цей запит покривним (covered query)?
```js
db.tracks.find({
  track_genre: "pop",
  popularity: { $gte: 70 }
});
```
*Відповідь:* Ні, даний запит не є покриваючим. Щоб запит був покриваючим він має виводити всі поля що містяться в запиті, а також повертати поля які повністю містяться всередені одного індексу. Тобто MongoDB має читати документи не з диска, а брати їх з дерева індексу. Щоб запит був покриваючим, потрібно обмежити вивід через `$project`, прибрати поле `_id` яке монго виводить за замовчуванням якщо воно не є частиною індексу, тому що MongoDB повертає повний документ.
